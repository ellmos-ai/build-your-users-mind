# build-your-users-mind — независимый ToM-модуль (предчувствие обратной связи)

> **What you mind is what you get.** Это **рецепт**, а не фреймворк. Каждая модель агента использует его для построения ToM-модели *своего* пользователя: оценка собственных данных → выделение паттернов принятия решений → поддержка файлов аватара → привязка к собственному файлу памяти/правил/системного промта.
>
> **Ядро = предчувствие обратной связи (feedback precognition / feedforward):** прогнозируйте обратную связь от пользователя ДО того, как она поступит; используйте ее в качестве управляющего сигнала в его отсутствие; оценивайте прогноз на соответствие реальности для улучшения качества.
>
> **Шаблоны:** `templates/` (файлы аватара), `scripts/` (конвейер), `TAXONOMY.md` (8 типов), `skills/swarm-operations/` (рой классификации). Существует **приватная эталонная реализация** (на основе логов автора), но она не поставляется в комплекте.
>
> **Теоретическая база:** *Prompt-Archaeology* (метод, таксономия в `TAXONOMY.md`) + исследования ToM (ToM-SWE arXiv 2510.21903; Persistent Memory & User Profiles 2510.07925).

## Основной принцип

LLM никогда не видят необработанные гигабайты логов. **Детерминированные скрипты сначала сокращают** данные до чистого корпуса пользовательских запросов, написанных человеком; только после этого **рой классификации** работает семантически.
Суть не в том, «какие промты», а в том, **«какое решение → какой результат → был ли пользователь удовлетворен».**

## 6 шагов

### 1. Определение источника (адаптер источника)
Найдите свои журналы взаимодействия. Различается для каждой модели → см. `SOURCE-ADAPTERS.md`.
Извлекайте **только настоящие, написанные человеком промты** (без результатов работы инструментов, системных напоминаний, внедренных хуков, кратких резюме сжатия контекста). Поля: `ts, project, session, text`.

### 2. Сокращение (детерминированное, без LLM)
- Фильтрация синтетических шагов, удаление дубликатов, агрегирование стандартных ответов и мелких подтверждений.
- **Связь с последующими действиями:** определение следующего шага (или шагов) пользователя для каждого промта как `outcome_signal` (praise | reissue | correction | abandon | none) → сигнал удовлетворенности.
- Расчет **`decision_score`** с помощью словаря принятия решений (исправление/предпочтение/правило/управление).
- **РЕДАКТИРОВАНИЕ (обязательно перед сохранением):** маскирование секретов/токенов/ключей/электронной почты, а в зависимости от пользователя — также **медицинских/налоговых данных/IP-адресов**. Конфиденциальные данные пользователя маскируются.

### 3. Классификация (рой, иерархический + стигмергия)
Таксономия из 8 типов **SP/NT/NM/NS/KO/BE/RA/MP** (определения в `TAXONOMY.md`) + `decision_kind` (preference / correction / rule / direction_change / approval / rejection / process / none) + `formulation_pattern` (характерная формулировка пользователя). Для больших корпусов: лидеры доменов (Sonnet) направляют исполнителей фрагментов (Haiku).

### 4. Создание файлов аватара
Структура 1:1, как в `templates/` (скопируйте шаблоны, замените `<USER>`/`<AGENT>`):
`WHAT-<USER>-SAID.md` (доказано) · `WHAT-WOULD-<USER>-SAY.md` (прогноз + уверенность) · `WHAT-I-DID-…md` + `MY-ACTIONS.txt` (журнал действий) · `WHAT-<USER>-SAID-ABOUT-…md` (полученные уроки) · `PROMPT-LOG` (вырезки и подсказки) · `METHODIK.md` (включая предупреждение о предвзятости) · `START.md` (цикл 0→4).

### 5. Привязка (критически важно!)
ToM-модель должна **действительно использоваться**, а не просто существовать:
- Короткое правило/указатель в **собственный файл памяти/правил/системный промт** агента (Claude: `CLAUDE.md`; Codex: `GPT.md`/`AGENTS.md`; agy: `GEMINI.md`; Kimi: `KIMI.md`) → указывает на цикл `START.md`. **Делайте его коротким** (указатель, без полного текста).
- **Правило приоритета:** проектные решения в `DECISIONS.md` имеют приоритет над сквозным аватаром.
- **Опциональные точки входа через команды (нюансы):** открывают доступ к циклу на трёх уровнях глубины плюс оркестратор — `read-my-mind` (прогноз, 0→2, без действий), `decide-like-me` (одно решение, 0→2, компонент рабочего процесса), `be-my-avatar` (действие, полный 0→4, только обратимые операции, журналирование), `avatar-orchestrator` (цепочка из множества решений, объединяет элементы 🔴/необратимые в один вопрос). Шаблоны в `templates/commands/`.
- **Версионированная привязка:** каноническим считается проектная копия цикла/скила; если агент также поставляет зарегистрированную копию, **побеждает версия с бо́льшим номером**, а более старая маршрутизируемая копия **заменяется** — проект первичен, реестр вторичен (без расхождений).

### 6. Поддержка (самообучение)
- **Эмпирическая база:** периодически перезапускайте скрипты (журналы все равно сохраняются) — **без хуков на каждый промт** (избегайте ловушки идемпотентности/множественной регистрации; пакетная обработка более надежна).
- **Время выполнения (под руководством):** записывайте предположения, сделанные от имени пользователя → файл (3); при получении обратной связи → файл (4) → обновляйте правила в (1), корректируйте уверенность в (2).

## Предвзятость и ограничения (всегда упоминать)
- **Молчаливое одобрение невидимо** → исправления переоцениваются, из-за чего аватар кажется «критическим».
- **Хрупкость ToM:** надежно в повторяющихся ситуациях, хрупко при новых/соперничающих вариациях → уровни уверенности, при 🔴 **передавать на следующий уровень вместо угадывания**.
- Идентификаторы доказательств формируются в результате синтеза LLM → перепроверяйте ключевые из них по исходному корпусу для критических решений.

## Принцип простоты
Единственная часть, специфичная для модели, — это **адаптер источника** (Шаг 1). Рецепт, таксономия, структура аватара и привязка универсальны. Не усложняйте.
